## 使用说明

> 不变量（Invariant）是系统在任何时刻都必须满足的条件。
> 与业务规则不同，不变量是"绝对不可违反"的系统约束。
> AI Agent 在设计方案时，必须确保不违反任何不变量。

## 全局不变量

### INV-001: 账户余额非负
- **描述**: 用户账户余额在任何时刻不得为负数
- **作用域**: 用户账户聚合
- **检查时机**: 每次扣款操作前
- **违反后果**: 事务回滚，操作失败

### INV-002: 订单状态单向流转
- **描述**: 订单状态只能按照预定义的状态机流转，不可逆向
- **状态机**:

```
待支付 → 已支付 → 待发货 → 已发货 → 已完成
↓ ↓ ↓
已取消 退款中 → 已退款 售后中 → 售后完成
```

- **合法转换表**:

| 当前状态 | 可转换到 | 触发条件 |
|---------|---------|---------|
| 待支付 | 已支付 | 支付成功回调 |
| 待支付 | 已取消 | 用户主动取消 / 超时未支付(BR-050) |
| 已支付 | 待发货 | 系统自动流转（支付成功即待发货） |
| 已支付 | 退款中 | 用户发起退款且商家未发货 |
| 待发货 | 已发货 | 商家填写物流单号 |
| 已发货 | 已完成 | 用户确认收货 / 超时自动确认(BR-052) |
| 已完成 | 售后中 | 用户发起售后且在售后期内(BR-060) |
| 退款中 | 已退款 | 退款到账 |
| 售后中 | 售后完成 | 售后处理完结 |

- **作用域**: 订单聚合根
- **违反后果**: 抛出 IllegalStateTransitionException，操作拒绝
- **⚠️ AI检查提示**: 任何涉及订单状态变更的需求，必须对照此状态机验证合法性

### INV-003: 库存不可为负
- **描述**: 任何商品的可用库存数量在任何时刻不得小于0
- **作用域**: 库存聚合
- **检查时机**: 库存扣减操作
- **违反后果**: 更新失败，返回库存不足

### INV-004: 支付金额与订单金额一致
- **描述**: 支付完成时，实际支付金额必须等于订单应付金额
- **作用域**: 支付上下文 ↔ 订单上下文
- **检查时机**: 支付回调处理
- **允许偏差**: 0（精确匹配，精度到分）
- **违反后果**: 支付异常，进入人工审核队列

### INV-005: 唯一性约束
- **描述**: 以下字段在系统内全局唯一
| 实体 | 唯一字段 | 说明 |
|------|---------|------|
| 用户 | phone | 一个手机号只能注册一个账号 |
| 用户 | email | 一个邮箱只能绑定一个账号 |
| 商品 | sku_code | SKU编码全局唯一 |
| 订单 | order_no | 订单编号全局唯一 |

## 性能约束

| ID | 约束描述 | 指标 | 备注 |
|----|---------|------|------|
| PC-001 | 核心API响应时间 | P99 < 200ms | 商品详情、订单创建 |
| PC-002 | 系统可用性 | 99.95% | 月度统计 |
| PC-003 | 订单创建峰值QPS | ≥ 5000 | 大促场景 |
| PC-004 | 数据库单表行数 | < 5000万 | 超过需分表 |

## 安全约束

| ID | 约束描述 | 规格要求 |
|----|---------|---------|
| SC-001 | 用户密码不可明文存储 | bcrypt加密，cost factor ≥ 12 |
| SC-002 | 敏感数据传输加密 | 全站HTTPS，TLS 1.2+ |
| SC-003 | API访问需鉴权 | JWT Token，过期时间2小时 |
| SC-004 | 敏感字段脱敏 | 手机号中间4位星号，身份证保留首末各4位 |

## 数据约束

| ID | 约束描述 | 规则 |
|----|---------|------|
| DC-001 | 金额精度 | 所有金额使用分为单位存储(Long)，展示时转换为元 |
| DC-002 | 时间格式 | 统一使用UTC时间存储，展示时按用户时区转换 |
| DC-003 | 软删除 | 核心业务数据不物理删除，使用 deleted_at 标记 |
| DC-004 | 数据保留 | 订单数据保留至少3年，日志数据保留至少1年 |
